# Odoo 17.0 Internationalization Rules

> Category: i18n | Target: Odoo 17.0 | Load with: MASTER.md + i18n.md

## Translation Markup

### Wrap user-facing strings with `_()`

**WRONG:**
```python
from odoo import models

class LibraryBook(models.Model):
    _name = "library.book"

    def action_borrow(self):
        if not self.borrower_id:
            raise ValidationError("A borrower must be selected.")  # Not translatable
```

**CORRECT:**
```python
from odoo import _, models
from odoo.exceptions import ValidationError

class LibraryBook(models.Model):
    _name = "library.book"

    def action_borrow(self):
        if not self.borrower_id:
            raise ValidationError(_("A borrower must be selected."))
```

**Why:** `_()` marks the string for translation extraction. Without it, the string is hardcoded in English and cannot be translated. Always import `_` from `odoo` and wrap every user-facing string.

### Use `_()` for dynamic messages with `%s` or `.format()`

**WRONG:**
```python
raise ValidationError(f"Book {self.name} is not available.")
```

**CORRECT:**
```python
raise ValidationError(_("Book %s is not available.") % self.name)
```

**Why:** f-strings cannot be extracted for translation because the full string is not a static literal. Use `_()` with `%s` placeholders. The translation system extracts the template string; the `%` substitution happens at runtime.

### Do NOT translate technical strings

**WRONG:**
```python
_logger.info(_("Processing book %s") % book.name)  # Log messages are not user-facing
state = fields.Selection([
    ("draft", _("Draft")),  # Selection labels are auto-translatable
])
```

**CORRECT:**
```python
_logger.info("Processing book %s", book.name)  # No _() for logs
state = fields.Selection([
    ("draft", "Draft"),  # Selection labels are auto-translated via field string mechanism
])
```

**Why:** Log messages are for developers, not users -- do not translate them. Selection field labels are automatically translatable through Odoo's field translation mechanism; wrapping them in `_()` is redundant and can cause double-translation issues.

## Translatable Fields

### Field `string` parameter is auto-translatable

**CORRECT:**
```python
name = fields.Char(string="Title", required=True)
# "Title" is automatically available for translation
```

**Why:** The `string` parameter of all fields is automatically registered for translation. You do not need to wrap it in `_()`.

### Use `translate=True` for user-editable translatable content

**WRONG:**
```python
description = fields.Text(string="Description")
# Content is stored in one language only
```

**CORRECT:**
```python
description = fields.Text(string="Description", translate=True)
# Content can be stored in multiple languages
```

**Why:** `translate=True` enables per-language storage of field values. When a user switches language, they see the translated version. Use this for fields where the content itself (not just the label) should be translatable: product names, descriptions, categories.

### Do NOT use `translate=True` on relational or computed fields

**WRONG:**
```python
publisher_id = fields.Many2one("res.partner", translate=True)  # Nonsensical
full_name = fields.Char(compute="_compute_full_name", translate=True)  # Problematic
```

**CORRECT:**
```python
publisher_id = fields.Many2one("res.partner", string="Publisher")
full_name = fields.Char(compute="_compute_full_name", store=True)
```

**Why:** `translate=True` only makes sense on `Char`, `Text`, and `Html` fields that store user-entered content. Relational fields store IDs. Computed fields derive values from other fields.

## .pot File Generation

### Generate translation template with `odoo-bin`

**CORRECT:**
```bash
./odoo-bin --modules=library_book -d mydb --i18n-export=library_book.pot --language=
```

**Why:** This extracts all translatable strings from Python code (`_()` calls), XML views (strings, labels), and field definitions into a `.pot` file. Place the `.pot` file in the `i18n/` directory.

### Directory structure for translations

**CORRECT:**
```
i18n/
    library_book.pot    # Template file (source strings)
    fr.po               # French translation
    es.po               # Spanish translation
    ar.po               # Arabic translation
```

**Why:** The `.pot` file is the template. Language-specific `.po` files contain the translations. Odoo loads `.po` files matching the user's language during module installation.

## XML Translations

### Strings in XML views are auto-translatable

**CORRECT:**
```xml
<form string="Library Book">
    <group string="Book Details">
        <field name="name"/>
    </group>
    <button name="action_borrow" string="Borrow" type="object"/>
</form>
```

**Why:** The `string` attribute in XML view elements (`<form>`, `<group>`, `<page>`, `<button>`) is automatically extracted for translation. No additional markup needed.

## Changed in 17.0

| What Changed | Before (16.0) | Now (17.0) | Notes |
|-------------|---------------|------------|-------|
| Translation markup | `_()` function | Same, unchanged | Always use `from odoo import _` |
| `.pot` generation | `--i18n-export` flag | Same, unchanged | Standard extraction process |
| `translate=True` | Available on Char/Text/Html | Same, unchanged | Only use on content fields |

## pylint-odoo Rules

| Rule | Description | Fix |
|------|-------------|-----|
| W8160 | Missing `_()` on user-facing string in Python code | Wrap the string with `_("...")` |

**W8160 detail:** This fires when a string passed to `raise`, returned to the UI, or used in a notification is not wrapped in `_()`. It does NOT apply to log messages, technical strings, or field selection labels.

## Common Mistakes

### Missing `_()` on user-facing strings

Error messages in `raise ValidationError(...)`, `raise UserError(...)`, and notification messages MUST use `_()`. Without it, the message is not translatable.

### Translating technical strings

Do not translate field names, XML IDs, Python variable names, log messages, or SQL queries. Only user-visible text needs `_()`.

### Hardcoded strings in QWeb templates

**WRONG:**
```xml
<t t-set="title">My Page Title</t>
```

**CORRECT:**
```xml
<t t-set="title">My Page Title</t>
<!-- QWeb strings are auto-extracted for translation -->
```

**Why:** QWeb template strings are automatically extracted for translation. No special markup is needed in templates -- just ensure the module's `.pot` file is regenerated after adding new strings.

### Using f-strings with `_()`

**WRONG:**
```python
raise UserError(_(f"Book {name} not found"))
```

**CORRECT:**
```python
raise UserError(_("Book %s not found") % name)
```

**Why:** f-strings are evaluated before `_()` sees them, so the translation system gets the already-interpolated string (e.g., `"Book Moby Dick not found"`) instead of the template. The `.pot` file cannot extract dynamic strings.
