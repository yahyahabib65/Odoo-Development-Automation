    **Step 1: Render Structural Files** -- Keep existing odoo-gen-utils render-module invocation exactly as-is.

    **NEW: Checkpoint 1 -- Review Generated Models and Security (CP-1)**

    Write this as a prose markdown section (NOT PLAN.md XML). The agent reading generate.md will act on these instructions:

    ```markdown
    ## Checkpoint 1: Review Generated Models and Security

    Present the following structured summary to the user:

    **Generated Files:**
    - Models: list each file in models/ with model name
    - Security: security/security.xml, security/ir.model.access.csv, security/record_rules.xml (if present)
    - Tests: list each file in tests/
    - Views: list each file in views/
    - Data: list files in data/
    - Other: __manifest__.py, __init__.py, README.rst

    **Key Decisions Detected:**
    - State fields: list models with state/status Selection fields
    - Computed fields: list models with compute= fields
    - Sequence fields: list models with sequence-type Char fields
    - Multi-company: list models with company_id Many2one
    - Security groups: {module_name}_user, {module_name}_manager
    - ACL summary: User (Read, Write, Create), Manager (Read, Write, Create, Unlink)

    Prompt: "Review the generated structural files above. Type 'approved' to continue to business logic generation, or describe what needs to change."

    **If user types 'approved':** Continue to Step 2.

    **If user describes changes:**
    1. Increment $RETRY_COUNT for CP-1
    2. If $RETRY_COUNT > 3: Print "Maximum regeneration attempts (3) reached for structural files. Proceeding to Step 2. You can manually edit the files or re-run generation." Continue to Step 2.
    3. Otherwise: Re-invoke odoo-scaffold agent with the original spec + user feedback to update spec.json. Then re-run Step 1 (render-module).
    4. After re-render, show difflib.unified_diff of ONLY the changed files (compare old vs new versions). Do NOT show unchanged files.
    5. Prompt: "Here's what changed. Type 'approved' to continue, or describe further changes."
    6. Loop back to the approval check.
    ```

    **Step 2: Wave 1 -- Model Method Bodies** -- Keep existing odoo-model-gen invocation exactly as-is.

    **NEW: Checkpoint 2 -- Review Business Logic (CP-2)**

    ```markdown
    ## Checkpoint 2: Review Business Logic

    Present the following to the user:

    **Business Logic Added:**
    For each model processed by odoo-model-gen:
    - Computed fields: list method names and what they compute
    - Onchange handlers: list method names and trigger fields
    - Constraint methods: list method names and what they validate
    - TODO stubs replaced: count per model

    Prompt: "Review the business logic above. Type 'approved' to continue to view and test generation, or describe what needs to change."

    **If user types 'approved':** Continue to Step 3.

    **If user describes changes:**
    1. Increment $RETRY_COUNT for CP-2
    2. If $RETRY_COUNT > 3: Print "Maximum regeneration attempts (3) reached for business logic. Proceeding to Step 3. You can manually edit the files or re-run the model-gen agent." Continue to Step 3.
    3. Otherwise: Re-invoke odoo-model-gen for the affected model(s) with: original spec + current model file content + user feedback. Agent rewrites the ENTIRE model file.
    4. After rewrite, show difflib.unified_diff of the changed model files only.
    5. Prompt: "Here's what changed. Type 'approved' to continue, or describe further changes."
    6. Loop back to the approval check.

    **Regeneration scope:** CP-2 rejection re-runs Wave 1 (model-gen) for affected models, then proceeds to re-run ALL of Wave 2 (view-gen reads completed model files).
    ```

    **Step 3: Wave 2 -- View Enrichment + Test Generation** -- Keep existing odoo-view-gen and odoo-test-gen parallel invocation exactly as-is.

    **NEW: Checkpoint 3 -- Review Views and Tests (CP-3)**

    ```markdown
    ## Checkpoint 3: Review Views and Tests

    Present the following to the user:

    **View Enrichments:**
    - Models with workflow buttons added: list model names and button counts
    - View files modified: list paths

    **Test Coverage:**
    - Test files generated/enriched: list paths
    - Test categories present: computed field tests, constraint tests, onchange tests, CRUD tests, access rights tests, workflow transition tests

    Prompt: "Review the views and tests above. Type 'approved' to continue to i18n extraction and commit, or describe what needs to change."

    **If user types 'approved':** Continue to Step 3.5.

    **If user describes changes:**
    1. Increment $RETRY_COUNT for CP-3
    2. If $RETRY_COUNT > 3: Print "Maximum regeneration attempts (3) reached for views/tests. Proceeding to i18n extraction. You can manually edit the files or re-run the agents." Continue to Step 3.5.
    3. Otherwise: Re-invoke odoo-view-gen and/or odoo-test-gen with: original spec + current file content + user feedback. Agents rewrite ENTIRE files.
    4. After rewrite, show difflib.unified_diff of the changed files only.
    5. Prompt: "Here's what changed. Type 'approved' to continue, or describe further changes."
    6. Loop back to the approval check.

    **Regeneration scope:** CP-3 rejection re-runs Wave 2 only (view-gen and test-gen). Wave 1 model files are NOT re-run.
    ```

    **NEW: Step 3.5: Extract i18n Strings**

    ```markdown
    ## Step 3.5: Extract i18n Strings

    Run the i18n extractor to generate the .pot file:

    ```bash
    odoo-gen-utils extract-i18n "$OUTPUT_DIR/$MODULE_NAME"
    ```

    This scans all Python files for _() calls and XML files for string= attributes, then writes the POT file to $MODULE_NAME/i18n/$MODULE_NAME.pot.

    If the command fails, report the error but do NOT block the commit. The .pot file is a quality enhancement, not a generation blocker.

    Known limitation: field string= parameter translations (e.g., fields.Char(string="My Field")) are not extracted by static analysis. This is expected for v1.
    ```

    **Step 5: Summary Report** -- Keep existing format but add i18n line:
    ```
    i18n: {count} translatable strings extracted to i18n/{module_name}.pot
    ```

    **Error Handling** -- Keep existing section.

    IMPORTANT: All checkpoint sections must be written as PROSE MARKDOWN INSTRUCTIONS, not PLAN.md XML. The orchestrating agent (which reads generate.md) is Claude -- it reads the instructions and acts on them. Do NOT use `<task type="checkpoint:human-verify">` syntax.

    IMPORTANT: REVW-05 (skippable checkpoints) is already handled by GSD's `workflow.auto_advance = true` in config.json. Do NOT add any skip logic. Checkpoints are always present in the workflow; the skip behavior is GSD's existing mechanism.
