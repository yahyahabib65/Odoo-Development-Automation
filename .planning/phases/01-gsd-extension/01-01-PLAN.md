Create the core extension files in the project root (which will be cloned to `~/.claude/odoo-gen/`):

**VERSION** — Contains `0.1.0` (single line, no trailing content).

**defaults.json** — Odoo-specific global default configuration. Structure:
```json
{
  "odoo": {
    "odoo_version": "17.0",
    "edition": "community",
    "output_dir": ".",
    "license": "LGPL-3",
    "author": "",
    "website": "",
    "api_keys": {
      "_comment": "API keys are resolved from environment variables. Never store actual keys here.",
      "openai": "$OPENAI_API_KEY",
      "github": "$GITHUB_TOKEN"
    }
  }
}
```
This file provides defaults that agents read at runtime. Project-specific overrides go in `.planning/config.json` under an `odoo` key (EXT-03). The `api_keys` section documents which env vars agents expect but NEVER contains actual secrets — values starting with `$` are resolved from environment variables at runtime by the Python utility package. Config resolution precedence: project `.planning/config.json` > global `defaults.json` > environment variables (for api_keys).

**install.sh** — The single-command installer. Must:
1. Start with `#!/usr/bin/env bash` and `set -euo pipefail`
2. Determine `SCRIPT_DIR` (directory where install.sh lives) using `cd "$(dirname "$0")" && pwd`
3. Define `ODOO_GEN_DIR="$SCRIPT_DIR"` (the cloned repo IS the extension dir)
4. **Check prerequisites** (each prints clear error + URL and exits 1 if missing):
   - GSD: `[ -d "$HOME/.claude/get-shit-done" ]` — if missing: "GSD not found. Install: npx get-shit-done-cc@latest"
   - uv: `command -v uv` — if missing: "uv not found. Install: https://docs.astral.sh/uv/#getting-started"
   - Python 3.12: `uv python find 3.12` — if missing: "Python 3.12 not found. Install: uv python install 3.12"
5. **Create Python venv**: `uv venv "$ODOO_GEN_DIR/.venv" --python 3.12`
6. **Install Python package**: `"$ODOO_GEN_DIR/.venv/bin/pip" install -e "$ODOO_GEN_DIR/python/"`  (use venv pip directly, not `uv pip` which needs activation)
   - Actually use: `VIRTUAL_ENV="$ODOO_GEN_DIR/.venv" uv pip install -e "$ODOO_GEN_DIR/python/"` (uv respects VIRTUAL_ENV)
7. **Create wrapper script**: Write `$ODOO_GEN_DIR/bin/odoo-gen-utils` — a thin bash wrapper that runs `exec "$ODOO_GEN_DIR/.venv/bin/odoo-gen-utils" "$@"`. Make it executable. This solves the venv path resolution issue (Pitfall 4 from research).
8. **Register commands** (per CONTEXT.md — "same pattern GSD uses"): Claude Code auto-discovers commands from `~/.claude/commands/<namespace>/` directories. GSD uses this pattern (its commands live in `~/.claude/commands/gsd/`). Register odoo-gen commands the same way: `mkdir -p "$HOME/.claude/commands/odoo-gen" && cp "$ODOO_GEN_DIR/commands/"*.md "$HOME/.claude/commands/odoo-gen/"`. This makes `/odoo-gen:*` commands available as slash commands in any AI coding assistant that supports Claude Code's discovery mechanism.
9. **Symlink agent files**: Per CONTEXT.md decision, agent files live in `~/.claude/odoo-gen/agents/` and are NOT copied into GSD. Create symlinks: `for f in "$ODOO_GEN_DIR/agents/"*.md; do ln -sf "$f" "$HOME/.claude/agents/$(basename "$f")"; done`. This keeps agents in the extension directory while making them discoverable by Claude Code.
10. **Write manifest** for tracking: Write `$HOME/.claude/odoo-gen-manifest.json` listing all copied files (for clean uninstall/update).
11. **Verify**: Run `"$ODOO_GEN_DIR/bin/odoo-gen-utils" --version` and check exit code.
12. **Print success message** with installed version (from VERSION file) and list of registered commands.

Use colored output (green for success, red for error, yellow for warnings). Define color variables at top: `RED='\033[0;31m'`, `GREEN='\033[0;32m'`, `YELLOW='\033[1;33m'`, `NC='\033[0m'`.

Create the `bin/` directory as part of this task (it will hold the wrapper script after install).

Do NOT hardcode any secrets or API keys. The defaults.json `api_keys` field contains env var references (e.g., `"$OPENAI_API_KEY"`) that are resolved at runtime — never actual key values. Per EXT-03, api_keys is a required config field alongside odoo_version, edition, and output_dir.

**agents/odoo-scaffold.md** (ACTIVE — full implementation):
```yaml
---
name: odoo-scaffold
description: Scaffolds a complete Odoo 17.0 module from a natural language description. Parses intent, generates module spec, renders Jinja2 templates, produces installable module.
tools: Read, Write, Bash, Glob, Grep

---
```
System prompt (in `<role>` tags) must instruct the agent to:
1. Accept user's natural language module description from `$ARGUMENTS`
2. Parse the description to infer: module technical name (snake_case), module title (human-readable), models (names, fields, types, relationships), views needed, dependencies
3. Present the inferred spec to the user in a clear table format and wait for confirmation
4. On confirmation, call `odoo-gen-utils render` via Bash tool for each template to generate the full module
5. Announce each generation phase: "Generating models... Generating views... Generating security... Generating tests... Generating demo data... Done!"
6. The agent must use `$HOME/.claude/odoo-gen/bin/odoo-gen-utils` (the wrapper script) for all Python utility calls
7. Output directory is the current working directory: `./module_name/`
8. After generation, list all created files
9. Reference the scaffold workflow: `@~/.claude/odoo-gen/workflows/scaffold.md`

Include Odoo 17.0 specifics in the prompt:
- Version format: `17.0.1.0.0`
- `<tree>` for list views (NOT `<list>` which is Odoo 18+)
- Inline `invisible`/`readonly` expressions (NOT `attrs` which is deprecated in 17.0)
- `license` is required in manifest
- One Python file per model (OCA convention)
- Full OCA directory structure
